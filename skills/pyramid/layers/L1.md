# L1 — Principles (기술 원칙)

## 추상화 경계

- **다룸**: 기술 스택 선택의 근거, 아키텍처 방향, 절대 제약, 비기능 요건의 우선순위
- **다루지 않음**: 구체적 모듈 이름·경계 (L2), 타입 시그니처 (코드 진실 소스), 비즈니스 정책이 아닌 의사결정 과정

L1은 "왜 이 기술/방향인가"의 문서. 모듈명이 나오면 L2로 분리.

## 파일

- `_docs/L1-principles/{id}.md` — **주제별 파일 분할이 기본** (예: `principles.md`·`constraints.md`·`scope.md` 또는 `stack.md`·`architecture.md`·`constraints.md`). 단일 파일은 주제가 진짜 하나뿐일 때만. 편집 locality·diff granularity·선택적 prior 로딩에 유리

## 템플릿

```markdown
---
layer: L1
id: <문서 id>
title: <원칙 주제>
---

# <제목>

## 배경 (L0 연결)

이 원칙이 L0의 어떤 부분에서 파생되는지 1~2문단.

## 원칙

- 원칙 1: 구체적 서술
  - 근거: 왜 이게 맞는가 (L0 성공 기준과의 연결)
  - 반대 사례: 이 원칙이 어길 수 있는 대안은? 왜 채택하지 않았는가
- 원칙 2: ...

## 절대 제약

깨지면 안 되는 것들 (보안, 성능, 호환성 등).

## 의도적 trade-off

포기한 것과 얻은 것. "이건 안 한다" 선언도 prior로서 중요.
```

## 대화 질문

- 이 선택의 대안은 뭐가 있었는가? 왜 배제했는가?
- 깨지면 안 되는 것 3가지는?
- 의도적으로 포기한 것은 무엇인가? (기능, 유연성, 속도)
- 6개월 뒤 이 원칙이 틀렸다고 판명되면 어떻게 회복하는가?

## 평가 기준 (improve 모드)

- **L0 연결**: 각 원칙이 L0의 어떤 부분에서 파생되는지 문서 내 명시 — 없으면 WEAK
- **선택 근거**: 각 원칙에 "왜 이게 맞는가"와 대안 언급 — 없으면 WEAK
- **모듈명 부재**: L2 모듈 이름이 L1에 나오면 FAIL (이관)
- **시그니처 부재**: 타입·함수 시그니처가 L1에 나오면 FAIL
- **Trade-off 명시**: `scope.md`(또는 동등 섹션)에 "의도적으로 안 하는 것" 3개 이상 — 없으면 WEAK
- **절대 제약 구체성**: 추상("안정적") 아닌 검증 가능한 서술("경유지 재입력 시 경로 동일")

## Gotchas

- "모범 사례"나 "일반론"만 나열된 L1은 잡음. 이 프로젝트에 고유한 판단이 없으면 제거
- L0 없이 L1 쓰면 공중에 뜬 원칙이 됨. 반드시 L0 근거를 함께
- 양쪽 다 옳은 원칙을 모두 나열하면 prior 품질 ↓. 선택한 쪽만 남기고 배제한 쪽은 "의도적 trade-off"로
