# L0 — Vision (프로젝트 정체성)

## 추상화 경계

- **다룸**: 존재 이유, 핵심 사상, 타겟 사용자, 성공 기준
- **다루지 않음**: 기술 스택, 아키텍처, 모듈 경계, API — 모두 L1/L2/L3

L0에서 기술이 나오면 잘못된 레이어. "이 프로젝트가 왜 존재해야 하는가"만 남긴다.

## 파일

- `_docs/L0-vision/vision.md` — 정체성·핵심 사이클·성공 기준
- `_docs/L0-vision/origin.md` — 배경·문제·계기·대상

```markdown
# 비전

## 프로젝트 정체성

이 프로젝트의 한 줄 정의와 핵심 사상.

## 핵심 사이클

사용자·에이전트가 반복하는 주요 워크플로우.

## 성공 기준

측정 가능한 기준 3개 이상. "~한 상태가 되면 성공"처럼 검증 가능하게.
```

## 대화 질문

- 이 프로젝트를 한 단어로 표현한다면?
- 기존 도구로 안 되는 것이 무엇인가?
- 성공 판단을 어떻게 할 것인가? (6개월 뒤 무엇이 보이면 성공인가)
- 이 프로젝트를 쓰지 않을 사용자는 누구인가? (범위 경계)

사용자의 1문장 답에 대해 "왜?", "구체적 예는?", "대안을 시도했는가?"로 파고들기.

## 작성 규칙

- 섹션에 가이드 주석(`<!-- ... -->`) 허용 (drafting 중엔 유용, 최종본에선 제거 권장)
- 모든 섹션에 실제 내용 채움 (공란 금지)
- 한국어, 간결·명확

## 평가 기준 (improve 모드에서 사용)

각 항목을 PASS/WEAK/FAIL로 평가, WEAK 이하에 구체 개선 제안.

- **구체성**: 정체성·핵심 사이클이 이 프로젝트 고유한가 — "최고의 X" 같은 일반론이면 WEAK
- **측정성**: 성공 기준 3개 이상이 검증 가능한가 (정량 또는 분명한 상태 변화) — 추상이면 FAIL
- **범위 경계**: 비대상·반대 사례가 명시됐는가 — 없으면 WEAK
- **기술 중립**: 기술 스택·모듈·API가 L0에 나오면 FAIL (L1/L2로 이관)
- **자기 참조 없음**: "이 문서는 …" 같은 메타 설명 — 있으면 WEAK
