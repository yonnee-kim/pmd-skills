# L2 — Spec (모듈 경계)

## 추상화 경계

- **다룸**: 모듈 목록, 각 모듈의 책임·경계, 모듈 간 상호작용, 에러 정책, 데이터 흐름
- **다루지 않음**: 타입·함수 시그니처 (L3), 내부 알고리즘 (L3 또는 코드)

L2는 "무엇이 어디에 있는가"의 문서. 시그니처가 등장하면 L3로 내려보낸다.

## 파일

- `_docs/L2-spec/{id}.md` — 주로 `modules.md`, `interactions.md`, `errors.md` 같은 분할. 초기엔 `spec.md` 단일 파일도 OK

## 템플릿

```markdown
---
layer: L2
id: <문서 id>
title: <spec 주제>
---

# <제목>

## L1 연결

L1의 어떤 원칙이 이 모듈 구성에 반영됐는지.

## 모듈

### <모듈명>
- 책임: 한 문장
- 공개 경계: 어떤 것을 외부에 노출하는가 (이름만, 시그니처는 L3)
- 의존: 어떤 다른 모듈·외부 시스템에 의존
- 금지: 이 모듈이 하지 않는 것

### <모듈명 2>
...

## 상호작용

모듈 간 호출/이벤트 흐름. 순서·방향이 중요한 부분만.

## 에러 정책

에러 경계, 복구 전략, 사용자에게 노출되는 방식.
```

## 대화 질문

- 이 모듈을 빼면 어떻게 되는가? (경계 검증)
- A가 B를 직접 호출 vs 이벤트로 통신 — 어느 쪽? 왜?
- 에러가 어디에서 삼켜지고 어디에서 전파되는가?
- 모듈 간 순환 의존이 있다면 그건 L2가 틀렸다는 신호

## Gotchas

- 모듈이 10개 넘으면 그룹핑 단위(서브시스템)를 먼저 정의
- "어떻게 구현하는가"가 들어가면 L3로 내려보내기
- L1 없이 L2 쓰면 임의적으로 쪼갠 것처럼 보임. 분해 근거를 반드시 L1과 연결
- L2의 모듈 목록이 L3 `implements` 파일 경로의 상위 구조와 일치하는지 가끔 확인
