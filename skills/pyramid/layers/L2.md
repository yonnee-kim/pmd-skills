# L2 — Spec (모듈 경계 + 코드 매핑)

## 추상화 경계

- **다룸**: 모듈 목록, 각 모듈의 책임·경계·의존·금지, invariants, 상호작용, 에러 정책, **코드 매핑(`implements:`)**
- **다루지 않음**: 타입·함수 시그니처 세부, 내부 알고리즘 — 모두 코드가 진실 소스

L2는 "무엇이 어디에 있는가 + 어떤 코드에 살고 있는가"의 문서. 구체 시그니처는 코드에서 읽는다.

## 파일

- **모듈별 파일 분할이 기본** — `_docs/L2-spec/modules/{module}.md` 파일 하나당 모듈 하나. 각 파일 frontmatter에 `implements:`로 코드 경로 매핑
- 상호작용은 `_docs/L2-spec/interactions.md`, 에러 정책은 `_docs/L2-spec/errors.md`로 분리
- 하나의 모듈은 여러 코드 파일에 걸쳐 있을 수 있음 — `implements:` 리스트로 나열

## 모듈 파일 템플릿

```markdown
---
layer: L2
id: <모듈 id>
title: <모듈명>
implements:
  - lib/<module>/<file>.ts
  - lib/<module>/<other>.ts
---

# <모듈명>

## 책임

한 문장으로 이 모듈이 담당하는 것.

## 공개 경계

외부에 노출하는 API 이름 리스트. 시그니처는 코드 참조 (진실 소스).

- `functionName1` — 한 줄 설명
- `functionName2` — 한 줄 설명

## 의존

다른 모듈·외부 시스템.

## 금지

이 모듈이 **하지 않는** 것 (경계 명확화).

## Invariants (선택)

구현이 바뀌어도 깨지면 안 되는 속성. 타입 시스템에 안 잡히는 것 위주 (예: "polyline은 생성 후 불변", "공유 해제된 엔티티는 조회 불가"). 전역 제약은 L1 constraints로.
```

## 상호작용 파일 템플릿

`interactions.md`:
```markdown
---
layer: L2
id: interactions
title: 모듈 상호작용
---

# 모듈 상호작용

## A ↔ B
A가 B를 어떻게 호출하는지, 데이터 흐름·순서·방향.

## C ← D
...
```

## 에러 파일 템플릿

`errors.md`:
```markdown
---
layer: L2
id: errors
title: 에러 정책
---

# 에러 정책

- **케이스 이름**: 어떻게 처리하는가. 복구 전략. 사용자 노출 방식.
```

## 대화 질문

- 이 모듈을 빼면 어떻게 되는가? (경계 검증)
- A가 B를 직접 호출 vs 이벤트로 통신 — 어느 쪽? 왜?
- 에러가 어디에서 삼켜지고 어디에서 전파되는가?
- 모듈 간 순환 의존이 있다면 그건 L2가 틀렸다는 신호
- 이 모듈이 진화하며 타입 시스템에 안 잡히는 invariant가 있나?

## Gotchas

- **`implements:` 누락 금지** — hook이 drift를 탐지할 수 없음. 모듈 파일 생성 시 반드시 포함 (`${CLAUDE_PLUGIN_ROOT}/shared/mapping-spec.md` 참조)
- **시그니처 복붙 금지** — L2에 타입 시그니처를 박으면 코드와 drift가 즉시 발생. API 이름만 나열
- 모듈이 10개 넘으면 그룹핑 단위(서브시스템)를 먼저 정의
- "어떻게 구현하는가"가 들어가면 제거 — 코드에서 읽기
- L1 없이 L2 쓰면 임의적으로 쪼갠 것처럼 보임. 분해 근거를 L1과 연결
