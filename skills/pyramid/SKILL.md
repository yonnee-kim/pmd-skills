# pyramid — 문서 prior 엔진

문서 피라미드(L0 Vision → L1 Principles → L2 Modules)를 작성·진화·주입하는 컴패니언. 절차 스크립트가 아니라 **원칙·템플릿·질문 제공자**.

**3 레이어 구조**: L0(왜) → L1(어떤 방향/제약/범위) → L2(무엇이 어느 코드에 있는가). 시그니처·구현 세부는 **코드(L4)가 진실 소스** — 문서에 복붙하지 않는다.

posterior(doc↔code 정합성)는 `pyramid-sync`와 Stop hook의 몫 — 여기선 다루지 않는다.

## 상태 판독 후 진입

프로젝트 상태를 Glob/Read로 파악하고 사용자 요청과 결합해 분기.

- `init` — `_docs/L0-vision/`·동등 경로 없음 → L0부터 대화로 채움
- `draft` — L0 있음, 사용자가 다음 레이어 작성 요청 → 상위 레이어 Read → 타겟 레이어 초안
- `improve` — 대상 문서 이미 존재, 사용자가 리뷰/개선 요청 → 원칙 대조, 질문으로 밀도 올림
- `distill` — 코드 먼저 존재, "이 코드에서 L2 모듈 계약 뽑아줘" 요청 → 코드 읽고 L2 모듈 파일 역생성, `implements` 채움. 신규 개발 중 탐색적 모듈(써봐야 모양 결정되는)에도 사용

사용자 명시가 있으면 그쪽이 우선. 분기 선택의 자신 없으면 사용자에게 확인.

## 레이어 참조

각 레이어 작성 시 해당 파일 Read — 템플릿·추상화 수준·작성 원칙이 거기 있음.

- L0: `layers/L0.md` (vision, origin)
- L1: `layers/L1.md` (기술 원칙·절대 제약·범위)
- L2: `layers/L2.md` (모듈 경계·책임·`implements` 매핑·상호작용·에러)

레이어 간 추상화 수준 위반이 가장 흔한 실수 — 각 레이어 파일 상단의 "추상화 경계" 섹션을 엄수.

## 작성 원칙

- **질문 우선, 추측 금지**: 각 레이어 작성 전 `layers/L*.md`의 "대화 질문"을 던지고 사용자 답을 받는다. 도메인·원칙·모듈을 모델이 지어내면 prior가 아니라 hallucination. init 모드에선 부모 문서가 없으므로 특히 엄수
- **Prior S/N**: 모든 문장이 load-bearing. 파생 가능한 것은 쓰지 말 것
- **캘리브레이션**: "항상/절대" 대신 "대개/보통". Opus 4.7은 뉘앙스를 다룬다
- **단일 진실의 원천**: 같은 내용을 두 곳에 쓰지 말 것. 시그니처·알고리즘은 코드에만
- **미해결은 미해결로**: 결정되지 않은 것을 결정된 것처럼 쓰지 말 것. `TODO(why)` 허용
- **매핑은 L2 모듈에만**: `implements:` frontmatter는 L2 모듈 파일에 — `${CLAUDE_PLUGIN_ROOT}/shared/mapping-spec.md` 준수
- **수직 슬라이싱 우선**: L2 모듈을 한 번에 전부 쓰지 말 것. 구현할 모듈만 L2 문서화 → 코드 → 다음 모듈. prior 과잉 회피

## 질문으로 prior를 밀도 있게

대화형 질문이 prior 품질을 좌우. 각 레이어 파일에 질문 프롬프트 포함. 사용자의 1문장 답이 나오면 더 파고들기 — "왜?", "대안은?", "성공 기준은?" 단, 지루해지면 멈추고 초안 제시 후 반복.

## 모드별 가이던스

### init
- CLAUDE.md·`.gitignore`는 명시 요청 시에만 생성 (프로젝트 레벨 결정)
- 한 번에 전체 레이어 만들지 말 것. L0만 먼저 → 사용자 리뷰 → L1 넘어가기

### draft
- 상위 레이어 Read → 추상화 수준 체크 → 초안 제시
- 초안은 구조(헤더)만 먼저 or 본문까지 — 사용자가 선호하는 쪽 확인
- 기존 파일이 있으면 덮어쓰기 금지 (`improve`로 전환 제안)
- L2는 모듈 단위 수직 슬라이싱 — 구현 순서 따라 모듈별로 문서화

### improve
- 대상 레이어 파일(`layers/L*.md`)의 **"평가 기준" 섹션**을 기준으로 PASS/WEAK/FAIL 채점 후 WEAK 이하 항목만 개선안 제시
- 구조 변경이 필요하면 "구조 변경 선호" 원칙에 따라 구조안부터 제시
- 수정은 한 섹션씩, 매번 확인

### distill
Scope를 먼저 판별 — 사용자 요청과 대상 규모로 결정.

**single-module** (기본): 한 모듈·작은 디렉토리의 코드 → L2 모듈 파일 역추출
- 대상 코드 파일·디렉토리 확인 후 Read
- 공개 export/class·함수 이름 + 암묵적 invariant 추출
- L2 모듈 파일 초안(`_docs/L2-spec/modules/{module}.md`) 작성, frontmatter `implements:`에 실제 파일 경로
- 시그니처·내부 알고리즘은 복붙하지 말 것 — 코드가 진실 소스
- 사용자 검토 후 상위(L1/L0) 연결 보강 요청

**whole-project** (기존 대형 프로젝트에 pyramid 처음 도입): bottom-up migration
1. 구조 스캔 — `git ls-files`·Glob로 디렉토리 트리·엔트리포인트 파악. 파일 수·언어·주요 폴더 요약을 사용자에게 제시
2. 모듈 경계 후보 제시 — 디렉토리·import 그래프에서 자연스러운 경계 추정해 사용자에게 확인 받음 ("이 폴더를 한 모듈로?", "이 둘을 분리?"). 확정 전까지 Write 금지
3. 모듈별 L2 스텁 생성 — 확정된 각 모듈에 대해 `_docs/L2-spec/modules/{module}.md` Write (`implements:` 포함). 시그니처·세부는 생략, 공개 경계 이름만
4. Cross-cutting 문서 — 기존 `firestore.rules`·`openapi.yaml` 등 있으면 `db_schema.md`·`api_schema.md`로 역추출
5. L1 원칙 역추출 — 코드에서 나타나는 기술 스택·패턴을 사용자에게 확인(추측 금지). 원칙·제약·범위를 대화로 확립
6. L0 vision·origin — 사용자 대화. 코드만으로 추정 금지
7. 이후 posterior(`pyramid-sync` + hook)가 drift 보정 — 초기 L2가 얇아도 자연스럽게 수렴

## Gotchas

- 레이어 경계 위반이 가장 흔한 문제: L1에 모듈명 등장 → L2로, L2에 타입 시그니처 → 코드로, L0에 기술 스택 → L1으로
- `implements:` 경로는 레포 루트 기준 상대 경로. glob·심볼 단위 금지 (`${CLAUDE_PLUGIN_ROOT}/shared/mapping-spec.md` 참조)
- 매핑 누락은 posterior가 drift를 탐지 못한다는 뜻 — L2 모듈 파일 생성 시 반드시 `implements` 포함 여부 체크
- "문서를 위한 문서"는 쓰지 말 것. 각 문서가 실제 prior로 기능하는지 — 모델이 이걸 읽고 다르게 행동할지 — 자문
