# L3 — Contract (모듈 계약)

## 추상화 경계

- **다룸**: 공개 API의 타입 시그니처·필드명·입출력·invariant, 구체 에러 타입, 제약 조건
- **다루지 않음**: 내부 구현 알고리즘, 성능 최적화 디테일

L3는 "어떤 형태로 호출되는가"의 문서. 코드가 맞춰야 할 계약. **모델은 이것만 보고 L4(코드)를 생성**한다.

## 파일

- 각 모듈당 1~N개 L3 문서. 경로는 프로젝트 컨벤션 따름 — 기본은 `docs/L3/{module}.md`
- `implements:` frontmatter로 코드 경로 매핑 (필수, `shared/mapping-spec.md` 참조)

## 템플릿

```markdown
---
layer: L3
id: <모듈 id>
title: <모듈명> 계약
implements:
  - src/<module>/<file>.ts
---

# <모듈명> Contract

## 개요

이 모듈이 외부에 노출하는 핵심 형태를 한 문단. L2 모듈 설명과 겹치지 않게 — L3는 시그니처 수준.

## 공개 API

### <함수/클래스 이름>

```ts
export function foo(x: X, y: Y): Z
```

- 의미: 한 문장
- 입력 제약: x는 비어 있지 않음, y > 0 등
- 출력 보장: Z는 sorted, 중복 없음 등
- 에러: 어떤 조건에서 어떤 에러를 던지는가

### <다음 API>
...

## Invariants

이 모듈이 유지하는 불변식. 구현이 바뀌어도 깨지면 안 되는 것.

## 에러 타입

```ts
class FooError extends Error { code: 'X' | 'Y' }
```

## 사용 예

```ts
const result = foo(...);
```
최소 1개 — 호출 맥락을 보여주는 수준으로.
```

## `implements` 필수

`shared/mapping-spec.md` 참조. 핵심:
- 레포 루트 기준 상대 경로
- 파일 단위만 (심볼·glob 불가)
- 누락 시 Stop hook이 drift를 탐지할 수 없음 — 즉 posterior 엔진이 작동하지 않음

여러 파일이 한 계약을 구성하면 리스트로 나열:

```yaml
implements:
  - src/auth/login.ts
  - src/auth/session.ts
```

## 대화 질문

- 이 함수의 입력에 null/빈 값이 올 수 있는가?
- 에러는 throw인가 Result<T, E>인가? 어느 쪽 컨벤션?
- 불변식이 깨진 상황을 어떻게 진단하는가?
- 이 계약이 바뀌면 어떤 다른 L3가 영향받는가?

## Gotchas

- "대략 이런 형태"는 계약이 아니다. 구체 타입·필드명 없으면 L3가 아니라 L2
- 예제가 없는 L3는 LLM이 잘못 해석하기 쉬움 — 최소 하나
- `implements` 경로가 존재하지 않아도 OK (doc-ahead 상태, posterior가 알려줌)
- 구현 세부(알고리즘·데이터 구조 선택)를 쓰고 싶다면 `_docs/L3/{module}-notes.md` 같은 **별도** 문서에 두되, `implements`는 달지 말 것
