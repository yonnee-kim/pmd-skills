# pyramid — 문서 prior 엔진

문서 피라미드(L0 Vision → L1 Principles → L2 Spec → L3 Contract)를 작성·진화·주입하는 컴패니언. 절차 스크립트가 아니라 **원칙·템플릿·질문 제공자**.

posterior(doc↔code 정합성)는 `pyramid-sync`와 Stop hook의 몫 — 여기선 다루지 않는다.

## 상태 판독 후 진입

프로젝트 상태를 먼저 Glob/Read로 파악하고 사용자 요청과 결합해 분기.

| 관찰                                                        | 모드          | 주된 행동                                     |
| ----------------------------------------------------------- | ------------- | --------------------------------------------- |
| `_docs/L0-vision/`·동등 경로 없음                           | `init`        | L0부터 대화로 채움                            |
| L0 있음, 사용자가 다음 레이어 작성 요청                     | `draft`       | 상위 레이어 Read → 타겟 레이어 초안           |
| 대상 문서 이미 존재, 사용자가 리뷰/개선 요청                | `improve`     | 원칙 대조, 질문으로 밀도 올림                 |
| 코드만 있고 L3 없음, 사용자가 "기존 코드 문서화" 요청       | `migration`   | 코드 읽고 L3부터 역설계, `implements` 채움    |

사용자 명시가 있으면 그쪽이 우선. 분기 선택의 자신 없으면 사용자에게 확인.

## 레이어 참조

각 레이어 작성 시 해당 파일을 Read — 템플릿과 추상화 수준, 작성 원칙이 거기 있음.

- L0: `layers/L0.md` (vision, origin)
- L1: `layers/L1.md` (기술 원칙·아키텍처 방향)
- L2: `layers/L2.md` (모듈 경계·책임·상호작용)
- L3: `layers/L3.md` (모듈 계약 + `implements` 매핑)

레이어 간 추상화 수준 위반이 가장 흔한 실수 — 각 레이어 파일 상단의 "추상화 경계" 섹션을 엄수.

## 작성 원칙

- **Prior S/N**: 모든 문장이 load-bearing. 파생 가능한 것은 쓰지 말 것
- **캘리브레이션**: "항상/절대" 대신 "대개/보통". Opus 4.7은 뉘앙스를 다룬다
- **단일 진실의 원천**: 같은 내용을 두 곳에 쓰지 말 것 (리서치 후 확인)
- **미해결은 미해결로**: 결정되지 않은 것을 결정된 것처럼 쓰지 말 것. `TODO(why)` 허용
- **매핑은 L3에만**: `implements:` frontmatter는 L3 계약에만 — `shared/mapping-spec.md` 준수

## 질문으로 prior를 밀도 있게

대화형 질문이 prior 품질을 좌우. 각 레이어 파일에 질문 프롬프트 포함. 사용자의 1문장 답이 나오면 더 파고들기 — "왜?", "대안은?", "성공 기준은?" 단, 지루해지면 멈추고 초안 제시 후 반복.

## 모드별 가이던스

### init
- L0부터 시작. vision.md, origin.md를 대화로 채움
- CLAUDE.md·`.gitignore`는 명시 요청 시에만 생성 (프로젝트 레벨 결정)
- 한 번에 전체 레이어 만들지 말 것. L0만 먼저 → 사용자 리뷰 → L1 넘어가기

### draft
- 상위 레이어 Read → 추상화 수준 체크 → 초안 제시
- 초안은 구조(헤더)만 먼저 or 본문까지 — 사용자가 선호하는 쪽 확인
- 기존 파일이 있으면 덮어쓰기 금지 (`improve`로 전환 제안)

### improve
- 레이어 원칙 대조하여 누락·약점 짚기
- 구조 변경이 필요하면 "구조 변경 선호" 원칙에 따라 구조안부터 제시
- 수정은 한 섹션씩, 매번 확인

### migration
- 코드 먼저 읽음(구조·엔트리포인트·핵심 타입)
- L3부터 역설계 — 코드 파일마다 어떤 계약이 암묵적으로 존재하는지 뽑아 `implements` 채움
- 그다음 L2 모듈 경계를 L3에서 역추출. L1/L0은 사용자와 대화로

## Gotchas

- 레이어 경계 위반은 가장 흔한 문제. L1에 모듈명이 나오면 경고, L2에 시그니처가 나오면 경고, L3에 원칙이 나오면 경고
- `implements:` 경로는 레포 루트 기준 상대 경로. glob·심볼 단위 금지 (`shared/mapping-spec.md` 참조)
- 매핑 누락은 posterior 쪽이 drift를 탐지하지 못한다는 뜻 — L3 생성 시 반드시 `implements` 포함 여부 체크
- "문서를 위한 문서"는 쓰지 말 것. 각 문서가 실제로 prior로 기능하는지 — 모델이 이걸 읽고 다르게 행동할지 — 자문할 것
